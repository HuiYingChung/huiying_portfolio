Little Cafe

A single-page e-commerce concept for a boutique handmade cafe — designed and built with semantic HTML, vanilla JavaScript, and a sticky shopping cart experience.

HTML5 CSS JavaScript Figma Responsive Design UX / Interaction Design
Overview

A small supply station for your busy day.

Little Cafe is a fictional micro-cafe concept built as the final project for GIT 417. The shop carries a tightly curated menu: two espresso drinks, two breakfast croissants, and two artisan cookies — six products total.

The project demonstrates semantic HTML5, modular CSS with custom properties, and vanilla JavaScript — including DOM manipulation, a dynamic shopping cart, regex-based form validation with inline errors, and a complete dark/light mode system. All code was written from scratch without frameworks.

Concept

What the requirements inspired.

The cost calculator requirement aligned perfectly with the mental model of a quick-service morning stop: pick a couple of items, see the total update instantly, check out. I wanted to build something that felt genuinely real — not a generic template exercise.

I deliberately chose a small, curated menu rather than a large catalog. The constraint strengthened the design: a focused shop section that matched the cafe's intimate, boutique identity.

The fixed 1280px layout also influenced the shop structure. Two columns — products on the left, cart always visible on the right — mimics holding a physical menu while watching your order tally up like a receipt.

Process

From wireframe to working code.

I built the site in deliberate phases, resolving each layer of complexity before moving to the next.

01

Discovery & Content Planning

Defined the shop concept, product lineup, brand voice, and full information architecture before writing any code. Created a content inventory of all six items with names, descriptions, and pricing.

02

Visual Design

Established the Sage Green + Warm Orange color system as CSS custom properties, enabling the dark/light mode toggle as a single class swap on body.dark-mode. Locked in type choices and spacing tokens.

03

Wireframing (Figma)

Built low-fidelity wireframes before touching code. The most critical decision: two-column shop layout with sticky cart sidebar, confirmed to keep the order total always visible — reducing checkout friction.

04

The "Zero Div" Challenge

Challenged myself to write semantic HTML with almost zero div tags — using <section>, <article>, <hgroup>, <aside>, <figure> throughout. This improved accessibility and made JS selection predictable.

05

JavaScript

Cart logic (add/remove/re-render), real-time total with 8% tax, hero fade carousel, contact form with conditional regex validation, customer object creation on submit, and localStorage theme persistence.

06

Responsive Design & Mobile UX

Rebuilt the layout for mobile — and identified a new problem in the process. See the next section.

UX Decision

The mobile cart problem — and how I solved it.

The two-column desktop layout works because both panels are always visible. On mobile, that collapses to a single column — and the cart gets pushed below the entire product list. A user could add items and have no feedback about where their order is, or how to check out without scrolling to the bottom.

🧾

Fixed cart icon

Stays anchored in the corner at all times — always reachable, regardless of scroll position.

🔢

Live badge counter

Updates every time an item is added or removed, giving instant visual confirmation of cart state.

⚡

Anchor jump to cart

Tapping the icon jumps to the Your Order section — no scrolling, direct path to checkout.

This pattern mirrors what users already know from mobile ordering apps. The anchor jump is a deliberate, practical choice: straightforward to implement, and meaningfully better than offering no shortcut at all.

Thinking ahead: A slide-up drawer would eliminate the back-and-forth of jumping to the cart and scrolling back up to keep ordering. That's the logical next iteration — but for this project, the anchor jump solved the core problem cleanly.

Process Note

Working with AI — openly.

I used AI tools (GitHub Copilot within VS Code, and Claude by Anthropic) as collaborative thinking and learning partners throughout this project. I want to be transparent about what that actually looked like:

Transparent

How AI was used in this project

  • Concept development — used AI as a sounding board to refine the brand name, tagline, and voice before any design work.
  • JavaScript learning — when I didn't yet understand cart state logic or conditional form validation, I asked AI to explain the concepts before writing the code myself. Like asking a senior developer to walk through the reasoning.
  • Debugging — described problems to AI and worked through solutions collaboratively. Understanding why something broke was always the goal, not just getting a fix.
  • Design decisions — used AI to list trade-offs (sticky cart vs. modal, color token structure for dark mode). All final calls were mine.
  • Mobile UX feature — the fixed cart icon and responsive rebuild were developed with AI assistance, after I identified the problem myself.

All code was written and typed by me. I believe using AI as a learning accelerator is an honest professional skill — and I'd rather document this openly than leave it out.

Reflection

What I learned.

What I struggled with

Cart state management was the steepest climb — pushing product objects, removing items by ID, and rebuilding the DOM correctly every time without duplicating nodes or losing state. Conditional form validation (phone field only validates if the user picks phone; email only if email) also took several iterations to clean up.

What I enjoyed most

The moment the Add button was clicked and the cart total updated instantly — that was genuinely satisfying. Watching the entire color system flip from warm cream to deep espresso with a single toggle was also a highlight. The Zero Div challenge gave me more confidence in writing meaningful, accessible HTML.

The bigger takeaway

Going back to make the site responsive — and realizing mid-process that layout wasn't the only thing that needed fixing on mobile — was the most useful learning. Asking "what does a user actually need here" rather than "does this display correctly" is the difference between a coded exercise and a considered design.

Next steps
